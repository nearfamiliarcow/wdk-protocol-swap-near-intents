# Architecture: `@tetherto/wdk-protocol-swap-1click`

## Overview

This package implements a WDK `SwapProtocol` that wraps the NEAR Intents 1Click API
(`https://1click.chaindefuser.com`). It enables cross-chain token swaps across 29+ blockchains
via a deposit-address model: the protocol fetches a quote (which includes a generated deposit
address), transfers tokens to that address on the source chain, and the 1Click backend routes
settlement through NEAR Intents.

The package is **chain-agnostic** — it depends only on `@tetherto/wdk-wallet` base interfaces
(`IWalletAccount`) and works with any WDK wallet implementation (EVM, TON, Solana, Tron, BTC).
There is no chain-specific import or logic inside this package.

---

## Package Identity

| Field | Value |
|---|---|
| Package name | `@tetherto/wdk-protocol-swap-1click` |
| Class name | `OneClickProtocol` |
| Extends | `SwapProtocol` from `@tetherto/wdk-wallet/protocols` |
| WDK protocol type | `swap` |
| Chain suffix | None — chain-agnostic |

### Why no chain suffix

WDK naming convention requires a chain suffix (e.g., `-evm`, `-solana`) only when the
implementation imports chain-specific account types or contains chain-specific logic. This
package depends exclusively on the base `IWalletAccount` interface methods (`transfer()`,
`sendTransaction()`, `getAddress()`), which all WDK wallet implementations provide. Adding a
suffix would be incorrect and would artificially limit how the package can be registered.

Compare to `@tetherto/wdk-protocol-swap-velora-evm`, which imports EVM-specific types and
calls EVM-specific methods — that package genuinely requires the `-evm` suffix.

## Changes to `wdk-wallet`

**File**: `wdk-wallet/src/protocols/swap-protocol.js`

Two optional fields are added to the `SwapCommonOptions` typedef:

```js
/**
 * @typedef {Object} SwapCommonOptions
 * @property {string} tokenIn - The address of the token to sell.
 * @property {string} tokenOut - The address of the token to buy.
 * @property {string} [to] - The address that will receive the output tokens.
 * @property {string} [destinationChain] - The destination blockchain for cross-chain swaps.
 * @property {string} [destinationAsset] - Provider-specific asset identifier for the output token.
 */
```

This is the **only** change to any existing WDK package. It is non-breaking: existing protocol
implementations (`VeloraProtocolEvm`, `AaveProtocolEvm`, etc.) receive options as a plain
object and ignore fields they do not use. No existing behavior changes.

The alternative — a new `CrossChainSwapProtocol` type — was rejected because it would require
new `getCrossChainSwapProtocol()` methods on `IWalletAccountWithProtocols`, new detection logic
in `wdk-manager`, and new test coverage in every wallet package, all to carry two optional
fields that fit naturally in the existing `SwapOptions` shape.

## Config

```js
/**
 * @typedef {Object} OneClickProtocolConfig
 * @property {string} sourceChain
 *   Required. The 1Click chain identifier for the source chain.
 *   Examples: "eth", "sol", "btc", "arb", "ton", "tron"
 *   This maps the WDK wallet registration to the 1Click asset namespace.
 *   Note: Tron's chain ID is "tron" (not "trx" which is the native token ticker).
 *
 * @property {string} [jwt]
 *   Optional. Bearer JWT token for authenticated API endpoints.
 *   Required at runtime for swap(). All quote requests (including dry: true) benefit from
 *   JWT — unauthenticated quotes incur an additional 0.2% fee on the quoted rate.
 *   Status and deposit/submit endpoints also require JWT.
 *   WARNING: Consumers must ensure the JWT is not expired before calling swap(). A present
 *   but expired JWT will pass the presence guard, the deposit will broadcast successfully,
 *   but POST /v0/deposit/submit and GET /v0/status will fail — leaving a committed on-chain
 *   deposit with no way to track it through the protocol. Validate the JWT `exp` claim
 *   before calling swap().
 *
 * @property {Array<{recipient: string, fee: number}>} [appFees]
 *   Optional. Application-level fees in basis points, forwarded to the 1Click quote request.
 *
 * @property {number} [slippageBps]
 *   Optional. Slippage tolerance in basis points. Default: 100 (1%).
 *   Sent as the `slippageTolerance` field in the quote request.
 *
 * @property {number} [deadlineMs]
 *   Optional. Quote validity window in milliseconds from time of quote request.
 *   Default: 600000 (10 minutes).
 *   Converted to ISO 8601 string for the `deadline` request field.
 *   Note: BTC users may need a longer deadline due to unpredictable confirmation times.
 *
 * @property {string} [baseUrl]
 *   Optional. Override the 1Click API base URL. Default: "https://1click.chaindefuser.com".
 *   Used for testing and staging environments.
 *
 * @property {Object} [depositTxOptions]
 *   Optional. Extra parameters spread into sendTransaction() for native asset deposits.
 *   Designed for BTC: { feeRate?: number, confirmationTarget?: number }.
 *   The protocol does not inspect these fields — they pass through unchanged.
 *   Note: These fields are NOT part of the base WDK `Transaction` typedef (`{ to, value }`).
 *   This is a deliberate type escape — BTC's `sendTransaction()` accepts these extra fields
 *   beyond the base interface. The spread is safe because unknown fields are ignored by
 *   non-BTC wallets.
 *
 * @property {number | bigint} [swapMaxFee]
 *   Optional. Inherited from SwapProtocolConfig. Maximum acceptable fee for the deposit
 *   transaction. If the estimated fee meets or exceeds this, swap() throws before
 *   broadcasting. Uses `>=` comparison to match Velora's behavior.
 */
```

`sourceChain` is required because the protocol cannot infer the 1Click chain ID from the WDK
wallet alone — the WDK label ("ethereum", "arbitrum") does not have a canonical mapping to
1Click's identifiers ("eth", "arb"). This mapping is explicit at registration time.

## 1Click API Schemas

### POST /v0/quote — Request

All fields are **required** unless marked optional.

```js
{
  dry: boolean,                    // true = price preview only, false = create deposit address
  swapType: string,                // "EXACT_INPUT" | "EXACT_OUTPUT"
  slippageTolerance: number,       // basis points (100 = 1%)
  originAsset: string,             // 1Click asset ID, e.g. "nep141:eth-0xdac1...omft.near"
  destinationAsset: string,        // 1Click asset ID
  amount: string,                  // SINGLE field — in smallest unit, as a STRING
                                   //   EXACT_INPUT: the exact input amount
                                   //   EXACT_OUTPUT: the exact desired output amount
  depositType: string,             // "ORIGIN_CHAIN" (always, for WDK)
  recipient: string,               // destination address on the output chain
  recipientType: string,           // "DESTINATION_CHAIN" (always, for WDK)
  refundTo: string,                // source address for refunds
  refundType: string,              // "ORIGIN_CHAIN" (always, for WDK)
  deadline: string,                // ISO 8601 timestamp, e.g. "2026-03-20T12:00:00.000Z"

  // Optional fields:
  appFees: Array | undefined,      // [{recipient: string, fee: number}] in bps
}
```

**Critical**: The `amount` field is a **string** representation of the amount in the token's
smallest unit (wei, satoshis, lamports, etc.). WDK bigints must be converted via `.toString()`.
There is no `amountIn`/`amountOut` split in the request — a single `amount` field is interpreted
based on `swapType`.

### POST /v0/quote — Response

```js
{
  correlationId: string,           // unique ID for support/debugging — ALWAYS log this
  timestamp: string,               // ISO 8601
  signature: string,               // cryptographic commitment to quote terms (for disputes)
  quoteRequest: { ... },           // echoed request

  quote: {                         // NESTED under .quote
    depositAddress: string | undefined,  // where to send funds (ABSENT if dry: true)
    depositMemo: string | undefined, // REQUIRED for Stellar deposits; absent otherwise
    amountIn: string,              // input amount in smallest unit (string)
    amountOut: string,             // output amount in smallest unit (string)
    minAmountIn: string,           // minimum acceptable input (important for EXACT_OUTPUT)
    maxAmountIn: string,           // maximum acceptable input (important for EXACT_OUTPUT)
    minAmountOut: string,          // minimum output after slippage
    deadline: string | undefined,  // HARD expiry — ABSENT if dry: true. Funds sent AFTER may be LOST
    timeWhenInactive: string | undefined,  // soft expiry — ABSENT if dry: true
    refundFee: string | undefined,  // fee charged on refund, in origin asset smallest unit
    timeEstimate: number,          // estimated processing time in seconds (after deposit confirms)
  }
}
```

**Important nesting**: Amounts are at `quoteResponse.quote.amountIn`, NOT `quoteResponse.amountIn`.

**Important optionality**: `depositAddress`, `deadline`, and `timeWhenInactive` are **absent**
when `dry: true`. The deadline guard in `swap()` must explicitly check for the field's presence
before comparing — see swap flow step 11.

### GET /v0/tokens — Response

Array of token objects:

```js
{
  assetId: string,                 // e.g. "nep141:eth.omft.near"
  decimals: number,                // token decimals
  blockchain: string,              // chain ID: "eth", "sol", "btc", etc. (NOT "chainId")
  symbol: string,                  // e.g. "ETH", "USDT"
  price: string,                   // USD price as string
  priceUpdatedAt: string,          // ISO 8601
  contractAddress: string | null,  // null for native assets, contract address for tokens
}
```

**Important**: The chain identifier field is `blockchain`, not `chainId`.

### GET /v0/status — Request

Query parameters:
- `depositAddress` (required)
- `depositMemo` (required for Stellar/memo chains)

### GET /v0/status — Response

```js
{
  status: string,                  // see status table below
  correlationId: string,
  updatedAt: string,               // ISO 8601
  quoteResponse: { ... },         // the original QuoteResponse (full echo)
  swapDetails: {                   // always present (non-null), but fields may be empty
    intentHashes: string[],
    nearTxHashes: string[],
    amountIn: string | undefined,         // actual amount deposited
    amountOut: string | undefined,        // actual amount received
    slippage: number | undefined,         // actual slippage
    originChainTxHashes: Array<{ hash: string, explorerUrl: string }>,
    destinationChainTxHashes: Array<{ hash: string, explorerUrl: string }>,
    depositedAmount: string | undefined,  // confirmed deposit amount
    refundedAmount: string | undefined,   // amount refunded (if applicable)
    refundReason: string | undefined,     // e.g. "PARTIAL_DEPOSIT"
  }
}
```

Note: `swapDetails` is always present (not nullable), but its nested fields may be undefined
or empty arrays in non-terminal states. Do not use `swapDetails?.field` — use `swapDetails.field`.

### POST /v0/deposit/submit — Request

```js
{
  txHash: string,                  // required — the deposit transaction hash
  depositAddress: string,          // required — from quoteResponse.quote.depositAddress
  memo: string | undefined,        // required if quoteResponse.quote.depositMemo was non-null
  nearSenderAccount: string | undefined, // required for NEAR-origin deposits (account.getAddress())
}
```

## Typedefs

### `OneClickSwapResult`

The standard `SwapResult` is extended with additional fields for deposit tracking and
dispute resolution. This is defined as a separate typedef to avoid violating the base
`SwapResult` contract — TypeScript consumers can access the extra fields without a cast.

```js
/**
 * @typedef {SwapResult & { depositAddress: string, depositMemo: (string|undefined), correlationId: string, quoteSignature: string, quoteResponse: Object }} OneClickSwapResult
 * @property {string} hash - The deposit transaction hash.
 * @property {bigint} fee - Always 0n. The 1Click fee is embedded in the exchange rate.
 *   The deposit gas cost is paid by account.transfer()/sendTransaction() internally
 *   but is not surfaced here — SwapResult.fee documents "the gas cost" of the swap
 *   operation itself, and the 1Click swap has no separable gas cost.
 * @property {bigint} tokenInAmount - The input amount (from quoteResponse.quote.amountIn).
 * @property {bigint} tokenOutAmount - The expected output amount (from quoteResponse.quote.amountOut).
 * @property {string} depositAddress - The deposit address for status polling.
 * @property {string} correlationId - Unique ID for support requests and debugging.
 * @property {string} quoteSignature - Cryptographic commitment to quote terms (for disputes).
 * @property {string | undefined} depositMemo - Memo value from the quote (for Stellar forward-compat).
 *   Undefined for non-memo chains. Include when calling getSwapStatus().
 * @property {Object} quoteResponse - Full 1Click QuoteResponse. Contains signature, deadline,
 *   correlationId, and all quote details. Required for dispute resolution.
 */
```

The `swap()` JSDoc return type is `Promise<OneClickSwapResult>`.
The `quoteSwap()` JSDoc return type is `Promise<Omit<SwapResult, 'hash'>>` (standard interface).

### `OneClickSwapStatusResult`

```js
/**
 * @typedef {Object} OneClickSwapStatusResult
```

## Asset Registry

**File**: `src/asset-registry.js`

### Problem

WDK addresses tokens by contract address (e.g., `0xdAC17F958D2ee523a2206206994597C13D831ec7`).
The 1Click API identifies tokens by a NEP-141 asset ID string
(e.g., `nep141:eth-0xdac17f958d2ee523a2206206994597c13d831ec7.omft.near`). A translation layer
is required.

The token list is fetched from `GET /v0/tokens`. It also tells us whether an asset is native
(no `contractAddress` field) vs. a token contract, which drives the deposit method selection.

### Design

The `AssetRegistry` is a module-level singleton, keyed by `baseUrl`. The registry is shared
across all `OneClickProtocol` instances that share the same `baseUrl`.

This is necessary because `wdk-manager` lazy-instantiates protocols — a new `OneClickProtocol`
instance is created on every call to `account.getSwapProtocol('1click')`. Without a
module-level cache, each call to `quoteSwap()` or `swap()` would trigger a fresh `GET /v0/tokens`
fetch. With the singleton, the fetch happens once per `baseUrl` per process.

```js
// module-level
const _registries = new Map() // baseUrl -> AssetRegistry instance

export function getRegistry(baseUrl, client) {
  if (!_registries.has(baseUrl)) {
    _registries.set(baseUrl, new AssetRegistry(baseUrl, client))
  }
  return _registries.get(baseUrl)
}

/** @visibleForTesting */
export function _clearRegistriesForTesting() {
  _registries.clear()
}
```

### Test isolation

The module-level singleton persists across tests in the same Jest worker. Without clearing,
tests that call `getRegistry()` will get cached state from previous tests. Use
`_clearRegistriesForTesting()` in `beforeEach` to ensure test isolation. Alternatively, use
`jest.resetModules()` + dynamic `await import(...)` per test.

### Native token sentinel

For native assets (ETH, BTC, SOL, etc.), the caller passes `'native'` as the `tokenAddress`
parameter. This is the **only** accepted sentinel — do not use chain-specific token symbols
like `'ETH'` or `'BTC'`.

The registry stores native assets with the key `"chain:native"` (from `contractAddress ?? 'native'`).
The `resolve()` method normalizes the caller's input with the same fallback.

```js
class AssetRegistry {
  // Map<"chain:contractAddress", { assetId: string, isNative: boolean, decimals: number, symbol: string }>
  _cache = new Map()
  _loadPromise = null

  async resolve(chain, tokenAddress) {
    await this._ensureLoaded()
    const key = `${chain}:${(tokenAddress ?? 'native').toLowerCase()}`
    const entry = this._cache.get(key)
    if (!entry) throw new Error(`OneClickProtocol: token not found in 1Click registry: ${key}`)
    return entry
  }

  async _ensureLoaded() {
    if (!this._loadPromise) {
      this._loadPromise = this._doLoad()
    }
    return this._loadPromise
  }

  async _doLoad() {
    const tokens = await this._client.getTokens()
    for (const token of tokens) {
      const key = `${token.blockchain}:${(token.contractAddress ?? 'native').toLowerCase()}`
      this._cache.set(key, {
        assetId: token.assetId,
        isNative: !token.contractAddress,
        decimals: token.decimals,
        symbol: token.symbol
      })
    }
  }
}
```

The `isNative` flag on each registry entry is authoritative for deposit method selection —
it comes from the 1Click token list itself, not from any chain-specific heuristic.

The field used is `token.blockchain` (not `token.chainId` which does not exist).

**Singleton client note**: `getRegistry(baseUrl, client)` ignores the `client` parameter if a
registry for that `baseUrl` already exists — it returns the cached instance with the original
client. This is safe because `GET /v0/tokens` is a public endpoint (no JWT required). If JWT
rotation occurs between protocol instantiations, the registry's client retains the original JWT,
but this has no effect on token fetches.

## Deposit Method

This is the most critical implementation decision in the package.

### The two code paths

All WDK wallets implement both `transfer()` (for token contracts) and `sendTransaction()` (for
native assets). The correct path is selected by whether `tokenIn` is a native asset or a token
contract, not by which chain is in use.

```js
async _deposit(depositAddress, tokenIn, amount, txOptions = {}) {
  const entry = await this._registry.resolve(this._config.sourceChain, tokenIn)

  if (entry.isNative) {
    // Native asset: ETH, BTC, SOL, TRX, TON
    // All WDK chains implement sendTransaction({ to, value }).
    // BTC-specific fields (feeRate, confirmationTarget) pass through via txOptions.
    return this._account.sendTransaction({
      to: depositAddress,
      value: amount,
      ...txOptions
    })
  } else {
    // Token contract: ERC-20, SPL, TRC-20, jetton
    // All non-BTC chains implement transfer().
    // BTC has no token layer so can never reach this branch.
    return this._account.transfer({
      token: tokenIn,
      recipient: depositAddress,
      amount
    })
  }
}
```

### Chain capability matrix

| Chain | `transfer()` | `sendTransaction({ to, value })` | Notes |
|---|---|---|---|
| EVM | Yes (ERC-20) | Yes (native ETH) | Standard |
| TON | Yes (jettons) | Yes (native TON) | Jetton caveat — see below |
| Solana | Yes (SPL tokens) | Yes (native SOL) | Standard |
| Tron | Yes (TRC-20) | Yes (native TRX) | Standard |
| BTC | **Throws "not supported"** | Yes (native BTC) | BTC is native-only |

BTC is the only chain where `transfer()` throws. BTC has no token layer, so it is structurally
impossible for a BTC `tokenIn` to be a token contract. The BTC wallet will never reach the
`transfer()` branch.

### TON jetton transfer caveat

TON jetton (fungible token) transfers are architecturally different from EVM ERC-20 transfers.
Sending a jetton requires routing the transfer message through the sender's own Jetton Wallet
contract — not directly to the recipient address. The WDK `wdk-wallet-ton` `transfer()`
implementation handles this internally. However, if the TON wallet implementation has bugs in
this routing, depositing a jetton to a 1Click deposit address will silently fail (funds stay
in the Jetton Wallet, deposit address never receives them). **Verify TON token swaps end-to-end
before shipping.**

### BTC-specific pass-through

BTC's `sendTransaction()` accepts extra fields: `feeRate` and `confirmationTarget`. The protocol
does not need to know about these — users configure them via `depositTxOptions` in the protocol
config, and they spread through `txOptions` transparently. The protocol remains chain-agnostic.

Note: The base WDK `Transaction` typedef only defines `{ to, value }`. The `depositTxOptions`
spread is a deliberate type escape — BTC's implementation accepts these extra fields beyond the
base interface. Unknown fields are ignored by non-BTC wallets, so the spread is safe.

### ERC-20 token approval

On EVM chains, `account.transfer()` calls the ERC-20 contract's `transfer(address,uint256)`
function directly — it does NOT go through an intermediary contract that would need `approve()`.
The deposit is a direct token transfer to the `depositAddress`, so no approval step is needed.

## Amount Handling

**This is a critical correctness concern.** Three representations are in play:

| Context | Type | Example (1000 USDT, 6 decimals) |
|---|---|---|
| WDK SwapOptions | `bigint` (base units) | `1000000000n` |
| 1Click API request | `string` (base units) | `"1000000000"` |
| 1Click API response | `string` (base units) | `"1000000000"` |

### Conversions

**WDK → 1Click request**: `BigInt(options.tokenInAmount).toString()`

**1Click response → WDK return**: `BigInt(quoteResponse.quote.amountIn)`

All amounts in the 1Click API are strings in the token's smallest unit (wei, satoshis,
lamports, etc.). Never pass raw `bigint` or `number` to the API — always `.toString()`.

### EXACT_OUTPUT deposit amount and price movement risk

For `EXACT_OUTPUT` swaps, the quote response includes `amountIn`, `minAmountIn`, and `maxAmountIn`.
The protocol deposits `BigInt(quoteResponse.quote.amountIn)` — the API's calculated input amount.

**Refund fee validation**: For small deposits (especially BTC), callers should verify that
`BigInt(quoteResponse.quote.amountIn) > BigInt(quoteResponse.quote.refundFee ?? '0')` before
depositing. If the deposit amount is smaller than the refund fee, a failed swap would leave
the user with zero — the entire deposit is consumed by the refund fee. This is particularly
relevant for BTC due to high on-chain transaction costs.

**Price movement warning**: `amountIn` is the estimated mid-price input. If market rates shift
between quote time and the deposit arriving on-chain, the actual processed amount could fall
below `minAmountIn`, triggering a refund. For maximum execution certainty on `EXACT_OUTPUT`
swaps, callers may want to deposit up to `maxAmountIn` (excess is refunded after swap
completes). The `quoteResponse` on the result contains both values for callers who need this.

## `swap()` Flow

```
1. Guard: account must implement write methods (not IWalletAccountReadOnly)
   - Check: typeof this._account.sendTransaction !== 'function'
   - WalletAccountReadOnly does NOT define sendTransaction or transfer.
     IWalletAccount adds both. This duck-type check is reliable without
     importing any chain-specific class, preserving chain-agnosticism.
   - throw Error('OneClickProtocol: swap() requires a non-read-only account.') if true

2. Guard: JWT must be configured
   - throw Error('OneClickProtocol: jwt is required for swap()') if absent
   - WARNING: This only checks presence, not expiry. Consumers must validate
     the JWT `exp` claim themselves before calling swap().

3. Resolve originAsset
   - registry.resolve(config.sourceChain, tokenIn) -> { assetId, isNative }

4. Resolve destinationAsset
   - If options.destinationAsset is provided, use it directly (provider-specific override)
   - Else resolve via registry using options.destinationChain ?? config.sourceChain, and tokenOut

5. Determine recipient
   - options.to if provided (cross-chain: recipient on destination chain)
   - else account.getAddress() (same-wallet swap)

6. Determine refundTo
   - Always account.getAddress() — not user-configurable
   - Refund always returns to the sending address on the source chain

7. Determine swapType and amount
   - If options.tokenInAmount is provided: swapType = "EXACT_INPUT", amount = BigInt(tokenInAmount).toString()
   - If options.tokenOutAmount is provided: swapType = "EXACT_OUTPUT", amount = BigInt(tokenOutAmount).toString()

8. Build quote request
   {
     dry: false,
     swapType,
     slippageTolerance: config.slippageBps ?? 100,
     originAsset: resolvedOriginAssetId,
     destinationAsset: resolvedDestinationAssetId,
     amount,                                              // string, smallest unit
     depositType: "ORIGIN_CHAIN",
     recipient,
     recipientType: "DESTINATION_CHAIN",
     refundTo,
     refundType: "ORIGIN_CHAIN",
     deadline: new Date(Date.now() + (config.deadlineMs ?? 600000)).toISOString(),
     ...(config.appFees ? { appFees: config.appFees } : {})
   }

9. POST /v0/quote → quoteResponse
   - Response is nested: amounts are at quoteResponse.quote.amountIn, etc.

10. Check depositAddress presence (API contract guard)
    - if (!quoteResponse.quote.depositAddress) {
        throw new Error('OneClickProtocol: quote response missing depositAddress on live quote')
      }
    - A missing depositAddress on a dry:false quote is an API contract violation.
      Without this guard, _deposit(undefined, ...) would call sendTransaction({ to: undefined }),
      whose behavior is wallet-dependent and potentially dangerous.

11. Check for memo deposit mode (memo guard)
    - if (quoteResponse.quote.depositMemo) throw Error(
        'OneClickProtocol: memo-based deposits are not yet supported. ' +
        'This deposit requires a memo/tag value which WDK TransferOptions does not support. ' +
        'Affected chains include Stellar (memo) and potentially XRP (destination tag).')
    - This guard blocks ANY chain that returns a depositMemo, not just Stellar.
      If XRP deposit addresses use the depositMemo field for destination tags, XRP-origin
      swaps will also be blocked until WDK supports memo/tag pass-through.

12. Check hard deadline from RESPONSE (not our request deadline)
    - The deadline field is ABSENT on dry:true responses, so this must check presence first:
      if (!quoteResponse.quote.deadline) {
        throw new Error('OneClickProtocol: quote response missing deadline — cannot proceed')
      }
      if (Date.now() >= new Date(quoteResponse.quote.deadline).getTime()) {
        throw new Error('OneClickProtocol: quote expired before deposit could be initiated')
      }
    - Without the presence check, `new Date(undefined).getTime()` returns NaN,
      and `Date.now() >= NaN` evaluates to false — silently bypassing the safety guard.

13. Determine deposit amount
    - depositAmount = BigInt(quoteResponse.quote.amountIn)

14. Optional: swapMaxFee guard
    - If config.swapMaxFee is set, estimate the deposit transaction cost
    - Use account.quoteTransfer() for tokens or account.quoteSendTransaction() for native
    - Throw if estimated fee >= config.swapMaxFee (uses >=, matching Velora's convention)

15. Execute deposit
    - _deposit(quoteResponse.quote.depositAddress, tokenIn, depositAmount, config.depositTxOptions ?? {})
    - Returns { hash, fee }

16. Best-effort POST /v0/deposit/submit
    - { txHash: hash, depositAddress: quoteResponse.quote.depositAddress }
    - If config.sourceChain === 'near', also include nearSenderAccount: account.getAddress()
    - Wrapped in try/catch — failure here does NOT fail the swap
    - The 1Click backend will detect the deposit independently; this is an optimization

17. Return OneClickSwapResult
    {
      hash,                              // deposit transaction hash (string)
      fee: 0n,                           // see "fee reporting" section below
      tokenInAmount: BigInt(quoteResponse.quote.amountIn),
      tokenOutAmount: BigInt(quoteResponse.quote.amountOut),
      depositAddress: quoteResponse.quote.depositAddress,
      depositMemo: quoteResponse.quote.depositMemo,  // undefined for non-memo chains
      correlationId: quoteResponse.correlationId,
      quoteSignature: quoteResponse.signature,
      quoteResponse                      // full response for debugging/disputes
    }
```

**JSDoc note**: The `swap()` method's JSDoc must document:
- Return type is `Promise<OneClickSwapResult>` (not bare `SwapResult`)
- `fee` is always `0n` and why
- The method returns on deposit broadcast, not settlement — callers must poll `getSwapStatus()`

## `quoteSwap()` Flow

```
1. Resolve originAsset (same as swap step 3)

2. Resolve destinationAsset (same as swap step 4)

3. Determine swapType and amount (same as swap step 7)

4. Build quote request (same as swap step 8, but with dry: true)
   - dry: true means no real deposit address is generated
   - The response will NOT include depositAddress, deadline, or timeWhenInactive
   - Works without JWT but incurs +0.2% fee penalty on quoted rate if unauthenticated
     (same JWT rule applies to all POST /v0/quote calls regardless of dry flag)

5. POST /v0/quote → quoteResponse

6. Return
   {
     fee: 0n,
     tokenInAmount: BigInt(quoteResponse.quote.amountIn),
     tokenOutAmount: BigInt(quoteResponse.quote.amountOut)
   }
   Note: SwapResult.hash is absent from quoteSwap() return value per SwapProtocol interface.
```

Note: `this._lastQuoteResponse` is NOT used. WDK creates a new `OneClickProtocol` instance
on every `account.getSwapProtocol('1click')` call, so any instance state set during `quoteSwap()`
would be lost by the time `swap()` is called on a different instance. Callers who need the
raw quote response should retain the return value from `quoteSwap()` themselves.

## `getSwapStatus(depositAddress, depositMemo)` Method

Settlement is asynchronous and can take 15+ minutes for cross-chain swaps. `swap()` returns
when the deposit transaction broadcasts — it does not poll for settlement. This matches the
implicit WDK contract: protocol methods return when the user's on-chain action is complete.

Polling is the caller's responsibility, via `getSwapStatus()`. This follows the precedent set
by `FiatProtocol.getTransactionDetail()`.

The `depositMemo` parameter is required for Stellar deposits (deferred, but the API signature
should accept it now for forward-compatibility).

**JWT requirement**: `getSwapStatus()` uses the `OneClickClient` constructed with the JWT from
config. The `GET /v0/status` endpoint requires authentication. If the protocol was constructed
without a JWT, or if the JWT has expired since construction, status calls will fail at the API
level. There is no guard on this method — the constraint is documented here and in the JWT
Authentication table. Consumers who poll for extended periods (e.g., 15+ minutes for BTC swaps)
must ensure their JWT remains valid throughout the polling lifecycle, not just at `swap()` time.

```js
async getSwapStatus(depositAddress, depositMemo) {
  const response = await this._client.getExecutionStatus(depositAddress, depositMemo)
  const terminal = ['SUCCESS', 'REFUNDED', 'FAILED'].includes(response.status)
  return {
    status: response.status,
    terminal,
    correlationId: response.correlationId,
    updatedAt: response.updatedAt,
    swapDetails: response.swapDetails   // always present, never null
  }
}
```

### Status values

| Status | Terminal | Meaning |
|---|---|---|
| `SUCCESS` | Yes | Output delivered to recipient |
| `REFUNDED` | Yes | Swap failed; funds returned to refundTo |
| `FAILED` | Yes | Unrecoverable error |
| `PENDING_DEPOSIT` | No | Awaiting on-chain deposit |
| `KNOWN_DEPOSIT_TX` | No | Deposit detected, not yet confirmed |
| `PROCESSING` | No | Swap executing via market makers |
| `INCOMPLETE_DEPOSIT` | No | Deposit below minimum; may receive more before deadline |

`INCOMPLETE_DEPOSIT` should be surfaced distinctly to callers — it means the deposit amount
was insufficient. The user can either send additional funds to the same deposit address before
the deadline, or wait for a refund at deadline. The `swapDetails.depositedAmount` field shows
how much was actually received.

## Fee Reporting

`SwapResult.fee` is documented in `SwapProtocol` as "the gas cost." The 1Click protocol bakes
its routing fee into the exchange rate — there is no separable fee amount returned by the API.
The gas cost of the deposit transaction IS available from `account.transfer()` /
`account.sendTransaction()` return value (`{ hash, fee }`), but it is not surfaced in the
`SwapResult` — the deposit gas fee is computed but intentionally discarded.

`fee: 0n` is the honest value here. Alternatives considered:

- Return the deposit gas cost: the `SwapResult.fee` field semantically documents "the gas cost"
  of the swap operation, not the deposit operation. The two are distinct in a deposit-address model.
- Return the 1Click protocol fee: the API does not return a discrete fee amount; it is embedded
  in the spread between `amountIn` and the market rate.

The quote response does include a `refundFee` field (fee charged if the swap is refunded).
This is available in the stored `quoteResponse` for callers who need it.

`0n` is returned and documented clearly in the JSDoc on both `swap()` and `quoteSwap()`.

## Quote Expiry and Deadline Handling

The 1Click quote response includes two time fields (both **absent** on `dry: true` responses):

- `timeWhenInactive` — soft expiry. The deposit address goes "cold" but funds sent here are
  still safe; processing just slows down.
- `deadline` — hard expiry. Funds sent to the deposit address **after this timestamp may be
  permanently lost**. This is a critical safety boundary.

The protocol checks the hard deadline at two points:

1. Presence check: `quoteResponse.quote.deadline` must exist (it is absent on dry responses
   and could theoretically be absent on malformed responses). If missing, throw immediately.
2. Expiry check: `Date.now() >= new Date(quoteResponse.quote.deadline).getTime()` — if true,
   throw before initiating the deposit.

Callers who build UIs should surface `timeWhenInactive` and `deadline` from the returned
`quoteResponse` to warn users.

BTC users are at elevated risk because Bitcoin confirmation times are unpredictable. A BTC
transaction can take 10+ minutes to confirm, potentially exceeding the quote deadline. Users
should use `deadlineMs` config to request a longer-lived quote, and should monitor
`getSwapStatus()` after broadcasting.

## EXACT_INPUT vs EXACT_OUTPUT

The 1Click API `swapType` is mapped from the `SwapOptions` discriminated union:

| SwapOptions variant | tokenInAmount | tokenOutAmount | 1Click swapType | `amount` field |
|---|---|---|---|---|
| Sell variant | provided | absent | `EXACT_INPUT` | `tokenInAmount.toString()` |
| Buy variant | absent | provided | `EXACT_OUTPUT` | `tokenOutAmount.toString()` |

For `EXACT_INPUT`:
- If the user sends more than `amountIn`, the excess is refunded.
- If the user sends less than `amountIn`, the full deposit is refunded at deadline.

For `EXACT_OUTPUT`:
- The quote response includes `minAmountIn`, `amountIn`, and `maxAmountIn`.
- The protocol deposits `BigInt(quoteResponse.quote.amountIn)` — the API's calculated input
  amount that achieves the requested output.
- Callers building UIs should show users the `minAmountIn`/`maxAmountIn` range from `quoteResponse`.
- See "Amount Handling" section for price movement risk warning.

## JWT Authentication

| Endpoint | JWT Required? | Notes |
|---|---|---|
| `GET /v0/tokens` | No | Public endpoint |
| `POST /v0/quote` | Recommended for all | Same JWT rule applies regardless of `dry` flag. Without JWT, a 0.2% fee penalty is applied to the quoted rate. |
| `GET /v0/status` | Yes | Requires authentication |
| `POST /v0/deposit/submit` | Yes | Requires authentication |

The protocol requires `jwt` in config for `swap()`. `quoteSwap()` works without JWT but the
quoted rate will be 0.2% worse — this means the price shown to the user during preview will
differ from the price they get on authenticated execution. Pass JWT for consistent pricing.

**JWT expiry warning**: The protocol checks for JWT _presence_ but does not validate expiry.
A present-but-expired JWT will pass the guard, the deposit will broadcast, and then
`submitDepositTx` and `getSwapStatus()` will fail — leaving a committed deposit that cannot
be tracked through the protocol (the 1Click backend still processes it, but the consumer
loses visibility). Consumers MUST validate JWT `exp` before calling `swap()`.

**JWT during polling**: The JWT must also remain valid throughout the polling lifecycle after
`swap()` returns. Cross-chain settlements can take 15+ minutes (BTC can take over an hour).
If the JWT expires during polling, `getSwapStatus()` calls will fail. Consumers should either
use long-lived JWTs or implement JWT refresh logic before each `getSwapStatus()` call. Note
that the JWT is frozen at `OneClickProtocol` construction time — to use a refreshed JWT, the
consumer must create a new protocol instance (which happens automatically via the WDK-level
`getSwapProtocol()` lazy instantiation if the config's JWT has been updated).

## Stellar — Deferred

The 1Click API returns a non-null `depositMemo` for Stellar-origin swaps. This requires the
user to include the memo value in the transfer transaction. WDK's `transfer()` and
`sendTransaction()` interfaces do not currently accept a memo field.

Stellar support is **deferred** until `TransferOptions` is extended with a `memo` field across
all relevant wallet implementations. The protocol detects `depositMemo` in the quote response
and throws a descriptive error rather than silently ignoring the memo:

```js
if (quoteResponse.quote.depositMemo) {
  throw new Error(
    'OneClickProtocol: memo-based deposits are not yet supported. ' +
    'This deposit requires a memo value which WDK TransferOptions does not support.'
  )
}
```

**Request-side requirement**: When Stellar is eventually enabled, the quote request must also
include `depositMode: "MEMO"` in the request body. Without it, the API may reject the request
or return a `SIMPLE` mode deposit address that cannot accept memo-bearing Stellar transactions.
The implementation should detect `sourceChain === 'xlm'` (or equivalent) and add `depositMode`
to the request. This is deferred along with the rest of Stellar support.

The `getSwapStatus()` method signature already accepts `depositMemo` for forward-compatibility.
The `submitDepositTx` request schema already includes `memo` for when this is enabled.

## Other Out-of-Scope Items

- **Cardano**: Partial support in the 1Click API. Do not advertise support until tested end-to-end.
- **Zcash shielded addresses**: Not supported by 1Click. Only transparent `t1`/`t3` addresses
  work. The protocol makes no attempt to detect or validate Zcash address types.
- **`customRecipientMsg`**: A NEAR-specific `ft_transfer_call` integration field. Misuse causes
  permanent fund loss. Not exposed in the WDK interface. If needed in the future, it should be
  an advanced escape-hatch config, not a standard option.
- **`connectedWallets` / `sessionId`**: 1Click API fields for wallet-connect integrations.
  Not relevant for WDK's programmatic model.
- **`FLEX_INPUT` / `ANY_INPUT` swap types**: The 1Click API supports these additional swap types
  but they do not map to WDK's `SwapOptions` discriminated union. Deferred unless a use case
  emerges.
- **XRP destination tags**: XRP addresses may require a destination tag. The 1Click API may
  surface this via the `depositMemo` field or encode it in the deposit address (X-Address format).
  The current memo guard (swap step 11) blocks any chain that returns `depositMemo`, which
  includes XRP if the API uses that field for tags. Callers passing XRP as `options.to` must
  include the full X-Address or tag-bearing classic address. No validation is performed.
- **`referral` field**: The 1Click API supports a `referral` string in the quote request for
  partner tracking/analytics. Not currently exposed in config. Add if needed for business tracking.
- **`quoteWaitingTimeMs` field**: Optional quote request field controlling how long the relay
  waits for market maker quotes. Omitting uses the API default. Setting to 5000ms may yield
  better rates on less-liquid pairs. Not currently exposed in config.
- **Same-chain swaps**: The 1Click API supports same-chain swaps (e.g., USDT→USDC on Ethereum).
  These are routed the same way as cross-chain swaps. The `destinationChain` option can be
  omitted for same-chain swaps (defaults to `sourceChain`).

## `one-click-client.js`

Thin fetch wrapper. No business logic. Handles:

- Base URL configuration
- JWT Bearer token injection (`Authorization: Bearer <jwt>`)
- JSON serialization / deserialization
- HTTP error detection (non-2xx status → throw with status code and body)

```js
export default class OneClickClient {
  constructor({ baseUrl = 'https://1click.chaindefuser.com', jwt } = {}) { ... }

  async getTokens() { ... }                                  // GET /v0/tokens
  async getQuote(requestBody) { ... }                        // POST /v0/quote
  async getExecutionStatus(depositAddress, depositMemo) { ... } // GET /v0/status?depositAddress=...&depositMemo=...
  async submitDepositTx(requestBody) { ... }                 // POST /v0/deposit/submit
}
```

The client does not implement retry logic. Retries are the caller's responsibility.

### Rate limits

The 1Click API has rate limits that are not publicly documented. The polling interval in the
usage example (30 seconds) is reasonable for a single swap. Production integrations with many
concurrent swaps should implement exponential backoff on status polling. If rate limits are hit,
the API returns a non-2xx response that the client will throw as an error — it is
indistinguishable from other API errors at the protocol level. Callers should catch and retry
with backoff rather than treating rate limit errors as terminal.

## `one-click-protocol.js` — Class Skeleton

```js
import { SwapProtocol } from '@tetherto/wdk-wallet/protocols'
import OneClickClient from './one-click-client.js'
import { getRegistry } from './asset-registry.js'

export default class OneClickProtocol extends SwapProtocol {

    super(account, config)
    if (!config.sourceChain) {
      throw new Error('OneClickProtocol: sourceChain is required')
    }

      baseUrl: config.baseUrl,
      jwt: config.jwt

    this._registry = getRegistry(
      config.baseUrl ?? 'https://1click.chaindefuser.com',
      this._client
    )
  }

  /** @returns {Promise<OneClickSwapResult>} */
  async swap(options) { ... }

  /** @returns {Promise<Omit<SwapResult, 'hash'>>} */
  async quoteSwap(options) { ... }

  /** @returns {Promise<OneClickSwapStatusResult>} */
  async getSwapStatus(depositAddress, depositMemo) { ... }

  async getSupportedTokens() { ... }

  /** @private */
  async _deposit(depositAddress, tokenIn, amount, txOptions = {}) { ... }

  /**
   * Resolves WDK token addresses to 1Click asset IDs for both origin and destination.
   * @private
   * @param {SwapOptions} options
   * @returns {Promise<{ originAssetId: string, destinationAssetId: string, isNativeIn: boolean }>}
   *   - originAssetId: resolved 1Click asset ID for tokenIn
   *   - destinationAssetId: resolved 1Click asset ID for tokenOut (or options.destinationAsset if provided)
   *   - isNativeIn: whether tokenIn is a native asset (drives deposit method selection in _deposit)
   */
  async _resolveAssets(options) { ... }

  /**
   * Builds the full POST /v0/quote request body from SwapOptions and config.
   * @private
   * @param {SwapOptions} options
   * @param {boolean} dry - true for quoteSwap(), false for swap()
   * @returns {Promise<Object>} The complete request body matching the POST /v0/quote schema.
   *   Calls account.getAddress() for recipient (if options.to is absent) and refundTo.
   */
  async _buildQuoteRequest(options, dry) { ... }
}
```

Note: No `_lastQuoteResponse` field. WDK creates a new instance per `getSwapProtocol()` call,
so instance state set during `quoteSwap()` would be lost by the time `swap()` is called.

## Registration: WDK-level vs Account-level

WDK supports two registration paths with different instantiation behavior:

- **`wdk.registerProtocol(blockchain, label, Protocol, config)`** — **Lazy instantiation**.
  A new `OneClickProtocol` instance is created on every `account.getSwapProtocol('1click')` call.
  This is the expected path. The "no instance state" invariant (invariant 11) applies here.

- **`account.registerProtocol(label, Protocol, config)`** — **Eager instantiation**.
  `new Protocol(account, config)` is called immediately at registration time. The instance is
  a singleton per account — the same instance is returned on every `getSwapProtocol()` call.
  Under this path, instance state DOES persist between `quoteSwap()` and `swap()` calls.

The architecture is designed for the WDK-level (lazy) path. The singleton `AssetRegistry` and
the absence of `_lastQuoteResponse` are both predicated on lazy instantiation. The protocol
will work under account-level registration, but consumers should not rely on instance state
persisting — the behavior differs between the two paths.

## Registration and Usage

```js
import WDK from '@tetherto/wdk'
import WalletManagerEvm from '@tetherto/wdk-wallet-evm'
import OneClickProtocol from '@tetherto/wdk-protocol-swap-1click'

const wdk = new WDK(seed)
  .registerWallet('ethereum', WalletManagerEvm, evmConfig)
  .registerProtocol('ethereum', '1click', OneClickProtocol, {
    sourceChain: 'eth',
    jwt: process.env.ONECLICK_JWT,
    slippageBps: 100
  })

const account = await wdk.getAccount('ethereum', 0)

// Lazy instantiation: new OneClickProtocol(account, config) happens here
const oneclick = account.getSwapProtocol('1click')

// Same-chain quote (dry run)
const quote = await oneclick.quoteSwap({
  tokenIn: '0xdAC17F958D2ee523a2206206994597C13D831ec7', // USDT on ETH
  tokenOut: '0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48', // USDC on ETH
  tokenInAmount: 1000_000000n // 1000 USDT (6 decimals)
})

// Cross-chain swap: USDT on ETH -> USDT on Arbitrum
const result = await oneclick.swap({
  tokenIn: '0xdAC17F958D2ee523a2206206994597C13D831ec7',
  tokenOut: '0xFd086bC7CD5C481DCC9C85ebE478A1C0b69FCbb9', // USDT on Arbitrum
  tokenInAmount: 1000_000000n,
  destinationChain: 'arb',
  to: '0x742d35Cc6634C0532925a3b844Bc9e7595f2bD18' // recipient on Arbitrum
})
// result.hash            — deposit tx hash on Ethereum
// result.depositAddress  — use to poll status
// result.correlationId   — for support requests
// result.quoteResponse   — full API response for debugging

// Poll until terminal
let status
do {
  await new Promise(r => setTimeout(r, 30_000))
  status = await oneclick.getSwapStatus(result.depositAddress)
} while (!status.terminal)

console.log(status.status) // 'SUCCESS' or 'REFUNDED' or 'FAILED'
```

## Key Invariants

1. `swap()` MUST return before settlement completes. It returns when the deposit transaction
   is broadcast, not when the swap settles. Callers who need settlement confirmation must poll
   `getSwapStatus()`. This contract MUST be documented in the `swap()` JSDoc.

2. `fee: 0n` MUST be returned in all `SwapResult` objects. The 1Click protocol fee is embedded
   in the exchange rate and cannot be isolated as a gas cost figure. This MUST be documented
   in the JSDoc on both `swap()` and `quoteSwap()`.

3. `_deposit()` MUST use `sendTransaction()` for native assets and `transfer()` for token
   contracts. This determination is made from the `AssetRegistry`, not from chain identity.

4. `submitDepositTx` MUST be fire-and-forget. Its failure MUST NOT propagate to the caller.
   The 1Click backend detects deposits independently.

5. The full `QuoteResponse` (including `signature`, `deadline`, `correlationId`) MUST be
   stored and returned on `OneClickSwapResult`. The `signature` is the cryptographic commitment
   to quote terms and is required for dispute resolution. The `correlationId` is required for
   support requests.

6. A deposit MUST NOT be initiated if the `deadline` field is absent from the quote response
   or if the current time has passed the deadline (`Date.now() >= new Date(deadline).getTime()`).
   Tokens sent to an expired deposit address may be permanently lost. The presence check is
   critical: `new Date(undefined).getTime()` returns `NaN`, and `Date.now() >= NaN` evaluates
   to `false` — silently bypassing the safety guard.

7. All amounts sent to the 1Click API MUST be strings (`.toString()`). All amounts received
   from the API MUST be converted to `BigInt()` before returning to the WDK caller.

8. The quote request MUST include all required fields: `dry`, `swapType`, `slippageTolerance`,
   `originAsset`, `destinationAsset`, `amount`, `depositType`, `recipient`, `recipientType`,
   `refundTo`, `refundType`, `deadline`.

9. The `swapMaxFee` comparison MUST use `>=` (not `>`) to match the convention established
   by `VeloraProtocolEvm`.

10. The read-only guard MUST use `typeof this._account.sendTransaction !== 'function'` —
    not `instanceof` checks, which would require importing chain-specific classes and break
    chain-agnosticism.

11. No instance state (`_lastQuoteResponse`, etc.) should be used to communicate between
    `quoteSwap()` and `swap()` calls — WDK creates a new protocol instance per
    `getSwapProtocol()` call.

12. A deposit MUST NOT be initiated if `depositAddress` is absent or empty in the live quote
    response. A missing `depositAddress` on a `dry: false` quote is an API contract violation
    and must throw rather than propagating `undefined` into the wallet layer.
